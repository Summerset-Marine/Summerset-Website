# SMC IMAGE SCRAPE & WIRING — PROMPT v2 (post-audit correction)

## WHY THIS PROMPT EXISTS

The v1 image scrape prompt ran and pulled 21 images onto 21 pages. It worked, but a full page-by-page audit of both **https://summersetmarine.replit.app/** (all 53 routes, enumerated from the live sitemaps) and **https://summersetmarine.com/** (the legacy Squarespace site, ~78 non-blog/non-shop URLs) turned up two problems:

1. **Reused images in the wrong places.** Several images are duplicated across pages in a way that makes distinct pages look identical to each other.
2. **~30 pages got zero photography** in v1 and are still logo-only, including the entire Permanent Piers hub, the Commercial pier page, all three service pages, all five resource pages, the blog (index + all 3 posts), and every `/projects`, `/testimonials`, and `/contact` sub-page under all three markets.

This prompt fixes both, using real source photography identified on the legacy site — not stock images.

---

## CRITICAL RULE: NO ACCIDENTAL DUPLICATION

An image may appear on more than one page **only** in these two sanctioned cases:
- A home-page teaser card may reuse the hero image of the specific page it links to (e.g., the "All Seasons HD" card on `/` may reuse the same photo as `/products/permanent-piers/all-seasons-hd`'s hero).
- A `/locations` card may reuse the same photo used on that location's market hub page (same place, same photo, intentional).

**Every other case of the same file appearing on two different pages is a bug.** In particular:
- A market **hub** page (`/markets/oconomowoc`, `/markets/door-county`) must **never** use the identical photo as one specific lake page underneath it. Lake Geneva already does this correctly (hub and `/geneva-lake` have two different photos) — match that pattern for Oconomowoc and Door County.
- A product **hub** page (`/products/lifts`, `/products/permanent-piers`) must never reuse the identical photo as one specific product-detail page underneath it.

Before wiring anything, print a list of every image filename you're about to use and how many pages reference it. Flag anything above the two sanctioned cases before proceeding.

---

## TWO DIFFERENT MECHANISMS — READ BEFORE STARTING

This site has two unrelated ways images get onto a page. Confusing them will waste a build cycle:

**A. Static hero/section images** — hardcoded in the React components (market pages, product pages, service pages, resource pages, the homepage). Fixing these is: download the file → save to `artifacts/summerset-marine/public/images/smc/[filename]` → update the `src` in the relevant `.tsx` file. No CMS involved.

**B. Sanity-driven content** — the `/markets/*/projects`, `/markets/*/testimonials`, `/markets/*/contact` (single testimonial + featured project), and every product page's "Project gallery photos" and "Customer testimonials" placeholder are populated by the `project` and `testimonial` Sanity schemas, fetched live via GROQ. **You cannot fix these by dropping a file into `public/`.** You have to create new Sanity documents.

> **BLOCKER — Noah needs to do this before Track B can run:** the `SANITY_API_TOKEN` currently in Replit Secrets is a **viewer (read-only) token**. Creating `project`/`testimonial` documents and uploading image assets requires a token with Editor/Write permission. Generate one at manage.sanity.io → summerset-marine project → API → Tokens, and either swap the secret temporarily or hand it to the agent for this run only. Blog post featured images (`blogPost.featuredImage`) have the same requirement — the 3 already-imported posts need an update, not a new document, but it's still a write operation.

Do Track A (static files) first — it's the bulk of the visual fix and has no blocker. Do Track B second, once the write token exists.
